# 功能开发插件 (Feature Development Plugin)

一个全面的结构化功能开发工作流，包含用于代码库探索、架构设计和质量审查的专业 Agent。

## 概述

功能开发插件提供了一种系统性的 7 阶段方法来实现新功能。它不是直接跳入代码，而是引导您了解代码库、提出澄清问题、设计架构并确保质量——从而产生与现有代码无缝集成的更好设计的功能。

## 理念

构建功能需要的不仅仅是编写代码。您需要：
- **在做出更改之前了解代码库**
- **提出问题**来澄清模糊的需求
- **深思熟虑地设计**然后再实现
- **在构建后进行质量审查**

此插件将这些实践嵌入到结构化工作流中，当您使用 `/feature-dev` 命令时自动运行。

## 命令：`/feature-dev`

启动包含 7 个不同阶段的引导式功能开发工作流。

**用法：**
```bash
/feature-dev 添加用户 OAuth 认证
```

或者简单：
```bash
/feature-dev
```

该命令将交互式地引导您完成整个过程。

## 7 阶段工作流

### 阶段 1：发现

**目标**：了解需要构建什么

**内容：**
- 如果功能请求不明确则进行澄清
- 询问您要解决什么问题
- 识别约束和要求
- 总结理解并与您确认

**示例：**
```
您：/feature-dev 添加缓存
Claude：让我了解一下您的需求...
        - 应该缓存什么？（API 响应、计算值等）
        - 您有什么性能要求？
        - 您有首选的缓存解决方案吗？
```

### 阶段 2：代码库探索

**目标**：了解相关的现有代码和模式

**内容：**
- 并行启动 2-3 个 `code-explorer` Agent
- 每个 Agent 探索不同方面（类似功能、架构、UI 模式）
- Agent 返回包含关键文件的综合分析
- Claude 读取所有识别的文件以建立深度理解
- 展示综合发现摘要

**启动的 Agent：**
- "查找与 [功能] 相似的功能并追踪实现"
- "为 [区域] 映射架构和抽象"
- "分析 [相关功能] 的当前实现"

**示例输出：**
```
找到类似功能：
- 用户认证 (src/auth/)：使用 JWT 令牌，中间件模式
- 会话管理 (src/session/)：Redis 支持，24小时过期
- API 安全 (src/api/middleware/)：限流，CORS

关键理解文件：
- src/auth/AuthService.ts:45 - 核心认证逻辑
- src/middleware/authMiddleware.ts:12 - 请求认证
- src/config/security.ts:8 - 安全配置
```

### 阶段 3：澄清问题

**目标**：填补空白并解决所有歧义

**内容：**
- 审查代码库发现和功能请求
- 识别未指定的方面：
  - 边缘情况
  - 错误处理
  - 集成点
  - 向后兼容性
  - 性能需求
- 以有组织的列表形式呈现所有问题
- **在继续之前等待您的答案**

**示例：**
```
在设计架构之前，我需要澄清：

1. OAuth 提供商：哪些 OAuth 提供商？（Google、GitHub、自定义？）
2. 用户数据：存储 OAuth 令牌还是仅用户配置文件？
3. 现有认证：替换当前认证还是同时添加？
4. 会话：与现有会话管理集成？
5. 错误处理：如何处理 OAuth 故障？
```

**关键**：此阶段确保在设计开始之前没有歧义。

### 阶段 4：架构设计

**目标**：设计多种实现方法

**内容：**
- 启动 2-3 个不同侧重点的 `code-architect` Agent：
  - **最小更改**：最小变更，最大复用
  - **清晰架构**：可维护性，优雅的抽象
  - **务实平衡**：速度 + 质量
- 审查所有方法
- 形成关于哪种最适合此任务的观点
- 展示带有权衡和推荐的比较
- **询问您喜欢哪种方法**

**示例输出：**
```
我设计了 3 种方法：

方法 1：最小更改
- 使用 OAuth 方法扩展现有 AuthService
- 将新 OAuth 路由添加到现有 auth 路由器
- 需要最少的重构
优点：快速、低风险
缺点：耦合 OAuth 到现有认证、更难测试

方法 2：清晰架构
- 新 OAuthService 与专用接口
- 分离的 OAuth 路由和中间件
- 重构 AuthService 使用通用接口
优点：清晰分离、可测试、可维护
缺点：更多文件、更多重构

方法 3：务实平衡
- 新 OAuthProvider 抽象
- 集成到现有 AuthService
- 最小重构、良好边界
优点：平衡复杂性和清晰度
缺点：仍有一些耦合

推荐：方法 3 - 给您干净的边界而无需过多重构，并且非常适合您现有的架构。

您想使用哪种方法？
```

### 阶段 5：实现

**目标**：构建功能

**内容：**
- **在开始之前等待明确批准**
- 读取上一阶段识别的所有相关文件
- 按照选择的架构实现
- 严格遵循代码库约定
- 编写清晰、文档完善的代码
- 随着进度更新待办事项

**注意：**
- 仅在您批准后才开始实现
- 遵循阶段 2 中发现的模式
- 使用阶段 4 中设计的架构
- 持续跟踪进度

### 阶段 6：质量审查

**目标**：确保代码简洁、DRY、优雅且功能正确

**内容：**
- 并行启动 3 个不同侧重点的 `code-reviewer` Agent：
  - **简洁/DRY/优雅**：代码质量和可维护性
  - **Bug/正确性**：功能正确性和逻辑错误
  - **约定/抽象**：项目标准和模式
- 整合发现
- 识别最高严重程度的问题
- **呈现发现并询问您想要的做法**：
  - 现在修复
  - 稍后修复
  - 照原样继续
- 根据您的决定处理问题

**示例输出：**
```
代码审查结果：

高优先级问题：
1. OAuth 回调缺少错误处理 (src/auth/oauth.ts:67)
2. 内存泄漏：OAuth 状态未清理 (src/auth/oauth.ts:89)

中优先级：
1. 可以简化令牌刷新逻辑 (src/auth/oauth.ts:120)
2. 考虑提取 OAuth 配置验证

所有测试通过。代码遵循项目约定。

您想怎么做？
```

### 阶段 7：总结

**目标**：记录完成的工作

**内容：**
- 标记所有待办事项完成
- 总结：
  - 构建了什么
  - 做出的关键决策
  - 修改的文件
  - 建议的后续步骤

**示例：**
```
功能完成：OAuth 认证

构建了什么：
- 支持 Google 和 GitHub 的 OAuth 提供商抽象
- 与现有 auth 集成的 OAuth 路由和中间件
- 令牌刷新和会话集成
- 所有 OAuth 流程的错误处理

关键决策：
- 使用 OAuthProvider 抽象的务实方法
- 与现有会话管理集成
- 添加 OAuth 状态以防止 CSRF

修改的文件：
- src/auth/OAuthProvider.ts (新)
- src/auth/AuthService.ts
- src/routes/auth.ts
- src/middleware/authMiddleware.ts

建议的后续步骤：
- 为 OAuth 流程添加测试
- 添加更多 OAuth 提供商（Microsoft、Apple）
- 更新文档
```

## Agent

### `code-explorer`

**目的**：通过追踪执行路径深入分析现有代码库功能

**专注领域：**
- 入口点和调用链
- 数据流和转换
- 架构层和模式
- 依赖项和集成
- 实现细节

**触发时机：**
- 在阶段 2 中自动
- 可在探索代码时手动调用

**输出：**
- 带文件:行引用的入口点
- 逐步执行流程
- 关键组件和职责
- 架构见解
- 必需阅读文件列表

### `code-architect`

**目的**：设计功能架构和实现蓝图

**专注领域：**
- 代码库模式分析
- 架构决策
- 组件设计
- 实现路线图
- 数据流和构建序列

**触发时机：**
- 在阶段 4 中自动
- 可为架构设计手动调用

**输出：**
- 发现模式和约定
- 带有理由的架构决策
- 完整组件设计
- 带有具体文件的实现图
- 带阶段的构建序列

### `code-reviewer`

**目的**：审查 bug、质量问题和项目约定

**专注领域：**
- 项目指南合规性（CLAUDE.md）
- Bug 检测
- 代码质量问题
- 基于置信度的过滤（仅报告高置信度问题 ≥80）

**触发时机：**
- 在阶段 6 中自动
- 可在编写代码后手动调用

**输出：**
- 关键问题（置信度 75-100）
- 重要问题（置信度 50-74）
- 带有文件:行引用的具体修复
- 项目指南引用

## 使用模式

### 完整工作流（推荐用于新功能）：
```bash
/feature-dev 为 API 端点添加限流
```

让工作流引导您完成所有 7 个阶段。

### 手动 Agent 调用：

**探索功能：**
```
"启动 code-explorer 追踪认证如何工作"
```

**设计架构：**
```
"启动 code-architect 设计缓存层"
```

**审查代码：**
```
"启动 code-reviewer 检查我最近的更改"
```

## 最佳实践

1. **对复杂功能使用完整工作流**：7 个阶段确保彻底规划
2. **深思熟虑地回答澄清问题**：阶段 3 防止未来混淆
3. ** Deliberately 选择架构**：阶段 4 给您选项是有原因的
4. **不要跳过代码审查**：阶段 6 在问题进入生产之前捕获它们
5. **阅读建议的文件**：阶段 2 识别关键文件——阅读它们以了解上下文

## 何时使用此插件

**适用于：**
- 涉及多个文件的新功能
- 需要架构决策的功能
- 与现有代码的复杂集成
- 需求有些不清楚的功能

**不适用于：**
- 单行 bug 修复
- 微小的更改
- 定义良好的简单任务
- 紧急 hotfix

## 要求

- 已安装 Claude Code
- Git 仓库（用于代码审查）
- 带有现有代码库的项目（工作流假设有现有代码可以学习）

## 故障排除

### Agent 运行时间过长

**问题**：代码探索或架构 Agent 很慢

**解决方案**：
- 对于大型代码库这是正常的
- Agent 在可能时并行运行
- 彻底性会带来更好的理解

### 太多澄清问题

**问题**：阶段 3 问太多问题

**解决方案**：
- 在初始功能请求中更具体
- 预先提供约束上下文
- 如果真的没有偏好说"您认为最好的是什么"

### 架构选项令人不知所措

**问题**：阶段 4 中有太多架构选项

**解决方案**：
- 信任推荐——它基于代码库分析
- 如果仍然不确定，请求更多解释
- 有疑问时选择务实选项

## 提示

- **在功能请求中具体化**：更多细节 = 更少的澄清问题
- **信任流程**：每个阶段都建立在前一个阶段之上
- **审查 Agent 输出**：Agent 提供关于您代码库的宝贵见解
- **不要跳过阶段**：每个阶段都有其目的
- **用于学习**：探索阶段教会您关于自己代码库的知识
